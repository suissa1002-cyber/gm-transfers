# Transfers — אפליקציית קליטת העברות בין סניפים

מערכת web שפותרת שתי בעיות בזרימת ההעברות של NewOrder:

1. **מלאי פנטום** — כשהעברה נסרקת+נשמרת בסניף **המקור**, הפריטים נספרים מיד במלאי סניף
   **היעד**, למרות שהם פיזית עדיין בדרך (בקופסה). נציג עלול להבטיח ללקוח "יש במלאי"
   בזמן שהמכשיר עדיין בטרנזיט. → **לוח "בהעברה"** מראה ליעד מה בדרך אליו.
2. **אין אימות קליטה** — אם נשלחו 30 מכשירים והגיעו 28, אין שלב סריקה-בקבלה, אז 2
   החוסרים בלתי-נראים (shrinkage). → **מסך קליטה** מחייב את היעד לסרוק כל מכשיר;
   ירוק = הכל הגיע, אדום = חסר סריאל X מתוך Y.

## למה זה אפשרי עכשיו
ה-API של NewOrder הוא **קריאה-בלבד**, אבל ב-04/06/2026 רפי (מפתח NewOrder) חשף ב-
`stock-operations` את שלושת השדות שצריך **גם בהעברות (operationType=5)**:
`branchId` (מקור), `receivingBranchId` (יעד), ו-`serials` בכל פריט. מאומת על פעולה 13678.

## ארכיטקטורה
- **Backend**: FastAPI. `poller.py` מושך `recent_transfers` (type 5) כל ~30ש' ומעדכן DB.
- **DB**: SQLite בפיתוח → Postgres ב-Render. טבלאות: `transfers`, `transfer_items`, `receive_scans`.
- **Frontend**: SPA יחיד, שדה סריקה ממוקד-אוטומטית (סורק = מקלדת+Enter). זהות סניף נשמרת
  ב-localStorage לכל דפדפן.
- **אירוח**: Render (אותו דפוס כמו `agents/uri/stock_watcher`) + GitHub Actions keep-alive.

## "אכיפה" ללא חסימה בקופה (כי ה-API קריאה-בלבד)
לא ניתן לחסום פעולה בקופה. האכיפה משלבת:
1. **הסלמה ניהולית בטלגרם** — פוש מיידי ליעד על העברה חדשה; תזכורת אם לא נקלט;
   הסלמה למנהלים מעל 24ש'; דוח יומי 09:00 של העברות פתוחות.
2. **האפליקציה = רשימת הפריקה** — המסך מראה מה אמור להיות בקופסה, כך שסריקה היא חלק
   טבעי מהפריקה ולא מטלה. כניסה דרך QR על פתק ההעברה פותחת ישר את מסך הקליטה.
3. **אישור שלילי** — גם אי-קליטה נרשמת ומדווחת ("נשלחו 4, מעולם לא נקלט"), כך ש-
   shrinkage שהיום נעלם — נחשף.
4. (עתידי) מלאי בהעברה מסומן "לא מאומת" לאורי/אתר עד שנקלט — תמריץ חיובי לסרוק.

## הרצה מקומית
```bash
cd agents/transfers
pip install -r requirements.txt
python3 main.py            # → http://localhost:8000  (PORT=8077 וכו' לשינוי פורט)
```
ה-DB נוצר אוטומטית (SQLite מקומי `transfers.db`). הסורק בחנות = מקלדת — שדה הסריקה
ממוקד-אוטומטית ומאזין ל-Enter.

## API
| Method | Path | תיאור |
|---|---|---|
| GET | `/health` | בריאות + מונים |
| GET | `/api/branches` | רשימת סניפים |
| GET | `/api/transfers?branch_id=N` | העברות פתוחות לסניף (לוח בהעברה) |
| GET | `/api/transfers/{op_id}` | פירוט העברה + פריטים |
| POST | `/api/scan` | `{branch_id, code, op_id?}` → קליטת סריקה |
| GET | `/api/admin/overview?days=7` | לוח ניהול — כל ההעברות בכל הסניפים (מי לא סרק/ממתין/נקלט) |
| POST | `/api/poll` | הרצת סבב poll ידני (debug) |

## קונסולת ניהול (סניף "אתר") — מוגנת בסיסמה
סניף **אתר** (id 5) הוא גם סניף קליטה רגיל (מעבירים אליו מלאי) וגם **קונסולת האופרציה**:
בבחירת "אתר" מופיע כפתור **🛠️ ניהול** שפותח טבלת-על של **כל** ההעברות בכל הסניפים —
מונים (ממתינים/חלקי/נקלטו/לא-נסרקו-24ש'+), מיון לפי דחיפות, ובאדום ההעברות ש"לא נסרקו",
כולל עמודת **"קלט ע״י"**. בחירת "אתר" וה-endpoint `/api/admin/*` דורשים `ADMIN_PASSWORD`
(header `X-Admin-Key`), כדי שסניף רגיל לא יראה את לוח הניהול.

## זיהוי עובד קולט
בקליטה ראשונה הנציג מקליד את שמו (נשמר בדפדפן, ניתן להחלפה). כל סריקה נרשמת עם שם
הנציג (`received_by`), ומוצג בפריט ("נקלט ע״י X") ובקונסולת הניהול.

## מוצרים לא-סידוריים (אוזניות/גיימינג) — לפי ברקוד
פריטים ללא סריאל (PS5 Dualsense, JBL/CMF/Xiaomi Buds וכו') מקבלים ב-ingest את **הברקוד**
של המוצר מהקופה (cached). הסריקה מתאימה לפי **סריאל או ברקוד** — כך שמוצר עם 3 יחידות
זהות נקלט ב-3 סריקות ברקוד.

## עיצוב
**Desktop-first ורספונסיבי** (99% מהשימוש בדסקטופ בחנות): לוח כרטיסים בגריד, מסך קליטה
דו-טורי (פאנל סריקה דביק + רשת פריטים), טבלת ניהול מלאה. מתכווץ נכון גם למובייל.

## פריסה (Render)
זהה לדפוס `agents/uri/stock_watcher`: Docker + Postgres + GitHub Actions keep-alive.
משתני סביבה ב-`render.yaml` (הסודות מוגדרים ב-Render UI). מומלץ Postgres (`DATABASE_URL`)
כי Render Free מוחק SQLite בכל deploy. ל-deep-links בהתראות הגדר `APP_BASE_URL`.
להתראות טלגרם הגדר `TELEGRAM_BOT_TOKEN` ו-`TELEGRAM_BRANCH_CHATS` (מיפוי סניף→chat).
