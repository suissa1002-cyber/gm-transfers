# תוכנית POC — מרכזייה עצמאית (FreePBX/Asterisk) שתחליף את 1com

> מסמך היתכנות + תכנון שלבי. נכתב 25/06/2026 לאחר שמיצינו את האינטגרציה מול 1com
> (REST API, WebRTC, softphone מוטמע ב-GreenOS). מטרה: להעריך עלות/מאמץ/סיכון אמיתיים
> למעבר למרכזייה בבעלותנו.

---

## 1. שורה תחתונה (TL;DR)

- **אפשרי לחלוטין.** 1com היא בעצם **Asterisk/FreePBX** מנוהל. אנחנו יכולים להריץ את אותה תוכנה בעצמנו (חינמית, קוד פתוח), עם הטלפונים הפיזיים הקיימים (SIP סטנדרטי).
- **התלות היחידה שלא נעלמת:** קישוריות לרשת הטלפון הציבורית (PSTN) — צריך **ספק SIP Trunk** + ניוד המספרים. כלומר מחליפים `1com (מרכזייה+קווים מאוגדים)` ב-`FreePBX שלנו + ספק-טראנק נפרד`.
- **היתרון הגדול:** כשאנחנו הבעלים — כל הטלאים שבנינו (פולינג CHANNELS, curl-בזרימה ששבר צלצול, המתנה לנטוויל, תקלת 202) **נעלמים**. AMI/ARI נותנים אירועים בזמן אמת → פופאפ מיידי, dialplan שלנו, CDR/הקלטות ישר מה-DB.
- **הסיכון העיקרי: אבטחה (toll fraud)** — לא הבנייה. Asterisk חשוף נתקף תוך שעות.
- **לא פרויקט סוף-שבוע.** מהלך תשתיתי עם אחריות תפעולית שוטפת.

## 2. ארכיטקטורה מוצעת

```
            PSTN / רשת טלפון ציבורית
                      │
            ┌─────────┴──────────┐
            │  ספק SIP Trunk     │   (אומניטלקום / PayCall / קונקט-טלקום)
            │  + המספרים שלנו     │   08-9477402, 08-9350202, 073-…
            └─────────┬──────────┘
                      │ SIP/TLS + SRTP
         ┌────────────┴─────────────┐
         │   FreePBX / Asterisk      │   VM בענן (Frankfurt/IL) — always-on
         │   IVR · תורים · נתב יום    │   firewall + fail2ban + TLS
         │   הקלטות · CDR · CallerID  │
         └───┬─────────┬─────────┬───┘
             │ SIP/UDP │ WSS     │ AMI/ARI (events + control)
     ┌───────┴──┐  ┌───┴─────┐  ┌┴──────────────┐
     │ טלפונים   │  │ Softphone│  │   GreenOS      │
     │ פיזיים    │  │ JsSIP    │  │ (web/worker)   │
     │ Fanvil×5  │  │ במחשב    │  │ פופאפ/CRM/CDR  │
     └──────────┘  └──────────┘  └───────────────┘
```

**שכבת WebRTC** (לסופטפון): Asterisk תומך native ב-WSS+DTLS-SRTP. נריץ nginx כ-reverse-proxy ל-`/ws` + TURN (coturn) — בדיוק כמו ש-1com עושים (`webrtc.1com.co.il`).

## 3. מיפוי רכיבים: 1com → FreePBX (הכל קיים)

| צורך נוכחי (1com) | מימוש ב-FreePBX/Asterisk |
|---|---|
| שלוחות 200-204 + סופטפון 205 | Extensions (PJSIP) — הטלפונים הפיזיים נרשמים אותו דבר |
| נתב יום / IVR ראשי + תת-נתבים | IVR module (גרפי ב-FreePBX) |
| תורים (סיטי 18078, הזמנות 18058) | Queues module |
| Caller ID יוצא לפי מספר ראשי | Outbound CID per-extension / outbound route |
| הקלטות שיחה | Call Recording (מובנה, נשמר ל-disk/S3) |
| CDR / היסטוריית שיחות | CDR DB (MySQL) — **גישה ישירה, בלי API** |
| הקפצת פופאפ / זיהוי מתקשר | **AMI/ARI events** — מיידי, event-driven (לא פולינג!) |
| Click2call / חיוג מהמחשב | ARI originate / AMI Originate |
| העברת שיחה | מובנה (REFER) — כבר עובד אצלנו |
| שעות פעילות / פתיח לילה | Time Conditions module |

**מסקנה:** אין רכיב אחד שאין לו מקבילה. FreePBX 17 (Asterisk 21/22) מכסה את הכל דרך GUI.

## 4. ספקי SIP Trunk בישראל (התלות שנשארת)

צריך ספק שייתן: טרמינציה (שיחות יוצאות ל-PSTN), origination (קבלת שיחות), ו**ניוד המספרים הקיימים**.

| ספק | הערות |
|---|---|
| **אומניטלקום** | wholesale VoIP + טרמינציה, מתאים לנפחים — [omnitelecom.co.il](https://www.omnitelecom.co.il/services/sip-trunk-solutions/) |
| **PayCall / trunk.co.il** | SIP Trunking ל-99.9% uptime, DID מקומיים + ניוד — [trunk.co.il](https://trunk.co.il/) |
| **קונקט טלקום** | הקצאת/ניוד מספרים בפריסה ארצית — [connect-t.co.il](https://www.connect-t.co.il/sip-trunk) |
| **סלקום / פרטנר עסקים** | ספקים גדולים (יקר יותר, יציב) — [cellcom](https://business.cellcom.co.il/land_line/phones/sip_trunk/item/sip-trunk/347) · [partner](https://www.partner.co.il/u/business_sip) |
| **IPTech / Teleco** | חלופות עסקיות קטנות יותר |

**מה לברר מכל ספק (לפני בחירה):**
1. תמיכה ב-SIP registration/IP-auth + **TLS/SRTP** (לא רק UDP פתוח).
2. **ניוד המספרים** שלנו (08-9477402, 08-9350202) — זמן + עלות + רגולציה.
3. תמחור: עלות חודשית לטראנק + לדקה (מקומי/נייד/חו"ל) + מספר ערוצים מקבילים (concurrent channels).
4. הגבלות anti-fraud מצד הספק (חסימת חו"ל by-default = שכבת הגנה).

## 5. אירוח (Hosting)

**אופציה A — ענן (מומלץ ל-POC):** VM קטן (2 vCPU / 4GB / 40GB) ב-Hetzner/DigitalOcean/AWS באזור קרוב (פרנקפורט). ~$10-25/חודש. יתרון: always-on, גיבוי, snapshot, IP קבוע.

**אופציה B — on-prem:** שרת קטן בחנות. יתרון: latency נמוך לטלפונים המקומיים. חיסרון: תלוי בחשמל/אינטרנט של הסניף, אחריות פיזית.

**המלצה:** ענן ל-POC ולפרודקשן. שקול on-prem רק אם יש דרישת latency/רגולציה.

**אחסון הקלטות:** דיסק מקומי + sync ל-S3/object-storage (זול, עמיד).

## 6. אבטחה — ⚠️ החלק הקריטי (toll fraud)

Asterisk חשוף לאינטרנט = יעד מס' 1. תוקפים מחייגים לחו"ל על חשבונך תוך שעות. **חובה:**

1. **Firewall** — לפתוח SIP/RTP **רק** ל-IP של ספק הטראנק + לטלפונים (לא 0.0.0.0). WebRTC על 443 בלבד.
2. **fail2ban** — חסימה אוטומטית אחרי ניסיונות רישום כושלים.
3. **TLS + SRTP** — הצפנת SIP+מדיה (לא UDP גלוי).
4. **הגבלת יעדי חיוג** — חסימת חו"ל/קווי פרימיום by-default; הרשאה מפורשת רק למה שצריך.
5. **תקרת עלות יומית** + התראות חריגה (כמו שכבר יש לנו ל-NewOrder).
6. **סיסמאות SIP חזקות** (אקראיות, לא 1234) + Responsive Firewall של FreePBX.
7. עדכוני אבטחה שוטפים ל-Asterisk/FreePBX.

> זה לא אופציונלי. מרכזייה לא-מאובטחת = חשבון טלפון של אלפי ₪ בלילה אחד. זו הסיבה המרכזית ש-1com "שווה את הכסף" — הם סופגים את האחריות הזו.

## 7. אינטגרציה ל-GreenOS — היתרון הגדול (AMI/ARI)

זה הטיעון הכי חזק בעד המעבר. כל הטלאים שבנינו מתחלפים בפתרון נקי:

| מה שיש לנו היום (workaround) | מה שיהיה (native) |
|---|---|
| פולינג CHANNELS כל 1.5ש' (latency, nodename) | **AMI/ARI events** — `Newchannel`/`DialBegin`/`BridgeEnter`/`Hangup` בזמן אמת → פופאפ **מיידי** |
| curl-בזרימה ששבר צלצול | dialplan שלנו — מריצים hook בלי לגעת ב-ring |
| זיהוי "נענה" מתוך CHANNELS (לא מדויק) | event `BridgeEnter` = נענה ודאי + מי ענה |
| CDR דרך proxyapi (cache, השהיה) | **שאילתה ישירה ל-CDR DB** (MySQL) — מיידי, מלא |
| הקלטות דרך API + uniqueid | קבצים אצלנו / DB — גישה ישירה |
| click2call דרך proxyapi | ARI `channels.originate` |
| תקלת 202 + המתנה ל-reprovision | אנחנו שולטים, אפס המתנה לספק |

**מימוש:** שירות קטן ב-GreenOS שמתחבר ל-ARI (WebSocket) → מקבל אירועים → מזין את הפופאפ/CRM. או Asterisk → webhook ל-endpoint שלנו על כל אירוע. **הכל event-driven, אפס פולינג.**

## 8. ניוד מספרים

- המספרים (08-9477402, 08-9350202, 073-3656401) מנוידים **מ-1com לספק הטראנק החדש** — תהליך רגולטורי (טופס ניוד, אישור בעלות, ~ימים-שבועות).
- **אסטרטגיה ללא-downtime:** קודם מקבלים מספר ניסיון חדש, בודקים הכל, ורק כשבטוח מניידים מספר אחד (למשל 073) כפיילוט, ואז את הראשי.
- ⚠️ ניוד = נקודת אל-חזור זמנית (כמה שעות חוסר זמינות בעת המעבר). לתאם לשעה שקטה.

## 9. תוכנית POC מדורגת (בלי סיכון)

**שלב 0 — הכנה (ימים):**
- VM בענן + FreePBX 17 מותקן + hardening בסיסי.
- פתיחת טראנק ניסיון מול ספק אחד (PayCall/אומניטלקום) + מספר זמני אחד.

**שלב 1 — POC במקביל ל-1com (שבוע):**
- רישום הסופטפון של GreenOS + 1-2 טלפונים פיזיים ל-FreePBX (במקביל, בלי לגעת ב-1com).
- בניית IVR בסיסי + תור + הקלטה.
- חיבור GreenOS ל-AMI/ARI → לוודא שהפופאפ קופץ event-driven.
- שיחות בדיקה דרך מספר הניסיון: נכנס/יוצא/IVR/הקלטה/זיהוי.

**שלב 2 — הקשחה ואמינות (שבועיים):**
- אבטחה מלאה (firewall/fail2ban/TLS/הגבלות יעד/תקרת עלות).
- מבחני עומס + יציבות 24/7 + ניטור (כמו Sentinel).
- שחזור IVR/תורים/שעות-פעילות במלואם.

**שלב 3 — פיילוט מספר אחד (שבוע):**
- ניוד מספר משני אחד (073) ל-FreePBX. תנועה אמיתית מבוקרת.

**שלב 4 — מעבר מלא:**
- ניוד הראשי + שאר השלוחות. ניתוק 1com.
